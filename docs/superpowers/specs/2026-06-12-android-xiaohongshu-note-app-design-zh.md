# Android 小红书笔记 App 设计文档

## 目标

构建一个个人 Android 应用，用于从照片和基本上下文信息生成小红书笔记。首个完整笔记类型为美食。应用设计应确保后续可添加其他笔记类型（如旅行、演唱会、博物馆、体育赛事等），而无需重建共享的草稿、审核和发布流程。

v1 发布模式为先审核后移交：应用生成笔记草稿，用户审核编辑后，通过 Android 分享将选中的图片和文字移交至小红书。最终发布确认在小红书内完成。

## 假设条件

- 应用为个人使用，运行在 Samsung、Xiaomi 等 Android 手机上。
- 小红书 App 已安装在同一手机上，且用户已登录。
- 可使用云端 AI 服务。
- v1 阶段无已确认的小红书个人账号公开发布 API。
- Android 分享可将图片和文字移交给小红书，但小红书决定接受多少内容以及内容在其编辑器中的位置。

## V1 用户流程

1. 从 Android 照片选择器添加 5-20 张美食照片。
2. 输入基本美食信息：菜品名称、餐厅名称、位置、口味备注、价格或评分（可选）、氛围描述。
3. 生成 AI 笔记变体。
4. 审核生成的标题、正文、话题标签和选中的照片。
5. 编辑文字、调整照片顺序，选择要使用的变体。
6. 点击发布移交。
7. 应用复制文字作为备用方案，并通过 Android 分享打开小红书，附带选中的图片和生成的文字。
8. 用户在小红书内完成最终确认。

## 架构

Android 应用使用 Kotlin 和 Jetpack Compose，采用简洁的分层结构：

- UI 层：Compose 页面和 ViewModel，涵盖草稿列表、创建表单、生成、审核编辑器、风格设置和发布移交。
- 数据层：Room 存储笔记草稿、选中的照片引用、风格偏好和生成结果。
- AI 客户端：Android 应用调用一个小型后端，而非直接调用 AI 服务商。
- 发布接口：一个发布行为的小型边界。v1 使用 Android 分享实现 `XiaohongshuSharePublisher`。未来如果官方 API 可用，可添加 `XiaohongshuApiPublisher`。

后端有一个主要的生成接口。它接收笔记类型、结构化元数据、选中的图片或图片引用，以及风格偏好，返回结构化的笔记变体。

### 后端

- **技术栈**：Python/FastAPI
- **AI 服务商**：Google Gemini（多模态，支持 base64 图片）
- **项目布局**：Monorepo — `android/` 和 `backend/` 目录在本仓库中

### 后端生成流程

1. 验证输入（5-20 张图片，必填字段）
2. 解析风格：首选风格排第一，然后从剩余风格池中选择 2 个
3. 发起 3 个并行的 Gemini 调用（每种风格一个），使用相同的提示模板 + 风格特定指令
4. 收集结果，验证 JSON schema，附加警告
5. 返回变体数组（2-3 个）。如果某个调用失败，返回成功的 2 个。
6. 如果全部 3 个失败 → 502 并附带错误详情。

## 数据模型

`NoteDraft`（笔记草稿）

- `id`
- `type`：v1 为 `food`；开放式设计 — 后续可扩展 `travel`、`concert`、`museum`、`sports_match` 或任意新类别
- `status`：`draft`、`generated`、`reviewed`、`shared`
- `photoUris`
- `selectedPublishPhotoUris`
- `title`
- `body`
- `hashtags`
- `createdAt`
- `updatedAt`

`FoodInfo`（美食信息）

- `dishName`（菜品名称）
- `restaurantName`（餐厅名称）
- `location`（位置）
- `tasteNotes`（口味备注）
- `priceOrRating`（价格或评分）
- `vibeNotes`（氛围备注）
- `personalNotes`（个人备注）

未来笔记类型扩展（开放式 — 以下为示例，非穷举列表）：

- `TravelInfo`：目的地、日期、行程、费用、贴士。
- `ConcertInfo`：艺人、场馆、座位/视野、歌单亮点、氛围。
- `MuseumInfo`：博物馆/展览名称、城市、最喜欢的作品、参观贴士。
- `SportsMatchInfo`：球队/选手、运动类型、场馆、比分、关键时刻、现场氛围。
- 任意其他笔记类型均可通过定义其元数据表和 AI 提示模板来添加。

草稿列表、AI 生成、审核编辑器和小红书移交在笔记类型间共享。仅基本信息表单和 AI 提示因类型而异。

## 风格默认值

应用支持默认笔记风格，用户无需每次都选择。

- 全局首选风格适用于所有笔记类型，除非被覆盖。
- 每种笔记类型可拥有独立的默认首选风格。
- 每个草稿可在生成前覆盖风格设置。
- 每次生成返回 2-3 个变体。
- 首选风格排在最前并预选。
- 用户可在移交前切换变体、编辑或重新生成。

风格解析逻辑：按类型偏好 → 全局默认（"all"）→ 硬编码回退（"casual_story"）。

美食 v1 风格：

- 休闲故事：个人化叙事风格。
- 实用推荐：口味、性价比、推荐菜品、适合人群。
- 小红书 punchy：更强的标题、短段落、搜索友好的话题标签。高能量但不过度夸大。
- 简洁/极简：自然语气，较少推广性语言。

## AI 生成行为

### Gemini 系统提示

```
你是一位小红书美食笔记作者。创作真实的中文社交媒体风格美食笔记。

使用简体中文写作，除非用户要求其他语言。

仅使用：
- 用户提供的结构化详情
- 从上传照片中的安全视觉观察（颜色、摆盘、分量外观、图片中展示的氛围）

不要编造：
- 具体价格、食材、奖项、营业时间、排队时间
- 餐厅人气、赞助、服务质量
- 用户未提供的位置详情
- 任何不可见或未提供的信息

如果用户未将笔记标记为赞助，请避免推广/广告式语言。

仅返回合法 JSON。不要 Markdown。不要解释。
```

### JSON 响应 Schema

```json
{
  "variants": [
    {
      "styleLabel": "Casual Story | Practical | XHS Punchy | Clean/Minimal",
      "title": "string",
      "body": "string",
      "hashtags": ["string"],
      "warnings": ["string"]
    }
  ]
}
```

生成 2-3 个变体。将用户的首选风格放在最前面。

### 风格定义

- **休闲故事**：个人叙事。"我去过……"的语气。可使用 emoji。像在跟朋友聊天。关注体验和感受。
- **实用推荐**：口味、性价比、推荐菜品、适合带谁来。可使用项目符号。有帮助且信息丰富。关注"值不值得去？"
- **小红书 Punchy**：强有力的标题钩子。带 emoji 标题的短段落。搜索优化的话题标签。高能量，但不夸大无依据的事实。
- **简洁/极简**：自然语气。少推广性。简单结构。诚实描述。"这就是它是什么样的"，而非"你必须去"。

AI 不得编造用户未提供的事实，如具体价格、食材、奖项、营业时间、排队时间或赞助信息。允许从照片进行视觉观察（颜色、摆盘、分量、氛围）。仅当草稿被明确标记为赞助时，才可生成赞助或推广性语言。

审核步骤在小红书移交前为强制步骤。

## 图片处理

美食 v1 支持每条笔记 5-20 张图片。后续笔记类型可能需要更大的图片集：

- 旅行：10-40 张。
- 博物馆：20-80 张。
- 演唱会：20-100+ 张。

v1 审核页面包含缩略图网格、照片排序控件和发布选中数量。所有照片在单次批量请求中上传（并行压缩，一次 POST /generate）。

图片在上传前在设备端压缩至 1024px JPEG 质量 85%。使用 Android 照片选择器（无需存储权限 — 系统 UI 组件，对用户选中的照片进行无权限只读访问）。

如果 Android 分享或小红书拒绝过多图片，应用保留草稿并显示手动回退路径。

未来笔记类型可添加相册分组、最佳照片选择或批量处理辅助功能。

## 错误处理

- AI 生成失败：保留草稿并显示重试。
- 单张图片压缩失败：跳过该图片并警告用户。
- 小红书未安装或不可作为分享目标：复制笔记文字并显示手动发布说明。
- Android 分享返回无确认：将笔记状态保持为 reviewed，除非用户标记为已分享。
- 草稿永不丢失。每个错误路径都保留 Room 中的本地草稿。

### 后端 HTTP 错误码

- 422：无效输入（缺少必填字段、图片数量错误）
- 413：图片过大或过多
- 429：Gemini 限流（附带 retry-after）
- 502：Gemini API 错误（记录日志，返回错误详情）
- 200 带警告：部分成功（返回 2 个变体，1 个失败）

## 测试

Android：

- 草稿状态变更、风格默认值解析和数据映射的单元测试。
- 创建、生成、审核和发布移交的 UI 测试。
- 在至少一台 Samsung 设备和一台 Xiaomi 系 Android 环境上进行手动设备测试（如有条件）。

后端：

- 有效美食输入的 API 测试。

- 美食图片数量边界的 API 测试。
- 使用示例美食笔记进行提示测试，验证 AI 不会编造无依据的细节。

## V1 范围

包含：

- 个人 Android 应用。
- 美食笔记创建。
- 美食草稿 5-20 张图片选择。
- 通过 FastAPI 后端 + Gemini 进行云端 AI 生成。
- 2-3 个生成变体，首选风格排在最前。
- 发布前审核和编辑。
- Android 分享移交至小红书。

不包含：

- 直接小红书 API 发布。
- 无障碍自动化。
- 公开多用户发布。
- 非美食笔记类型（旅行、演唱会、博物馆、体育等 — 已设计但 v1 不实现）。
- 小红书内的自动最终发布确认。
